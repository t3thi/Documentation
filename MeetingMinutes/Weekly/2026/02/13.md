## Topic 1: Merged Copy Patches, dbdoctor, and Free Mode Test Strategy

The meeting began with André congratulating Eric on recently merged patches around copy handling. Eric explained that the merged work would directly benefit a current client project and that he had already contacted Lolli about the related [`EXT:dbdoctor` PR #171](https://github.com/lolli42/dbdoctor/pull/171). The PR refers to the copy-process patches, including André's earlier [patch #84237](https://review.typo3.org/c/Packages/TYPO3.CMS/+/84237) and the already merged [copy content patch #92580](https://review.typo3.org/c/Packages/TYPO3.CMS/+/92580).

Eric also reported that he had asked Lolli about the remaining [free-mode content patch #92585](https://review.typo3.org/c/Packages/TYPO3.CMS/+/92585). The question was how the required Free Mode test data should be represented in TYPO3 Core's functional data-scenario tests.

Eric presented two possible strategies for testing Free Mode content copies. One option was to keep a dedicated import file for Free Mode scenarios. The other option was to add a Free Mode content element directly to the standard default import data and update all affected expected-result CSV files. The second approach was already technically prepared and all tests were green, but it touched more than 300 files.

During the review, André noticed that the functional data-scenario tests already contain a `Regular/FreeMode` area. Eric realized that the prepared broad patch had missed this existing structure. The team concluded that the new tests should be moved into that existing Free Mode-specific scope instead of modifying the default dataset globally.

The team then reverted the broader direction and explored how to move the three new copy tests into the existing Free Mode test classes and CSV files. Eric and André confirmed that the tests needed to be registered in the concrete `ActionTest` classes and that the expected CSV data had to be adapted for the existing Free Mode dataset.

## Topic 2: DataHandler Filtering and Review Readiness for Free Mode Copy Operations

The team then reviewed the actual `DataHandler` change in [patch #92777](https://review.typo3.org/c/Packages/TYPO3.CMS/+/92777) for copying Free Mode content. The patch adds handling for non-default language records that have no connection to a default-language parent. The goal is to prevent such records from being copied into a target where their language is not available.

Eric and André discussed whether the condition that checks for missing default-language references was necessary. They agreed that, even if the backend workflow normally reaches the copy method only for standalone non-default language records, the explicit checks improve readability and protect direct `DataHandler` usage. The code therefore keeps the checks that identify language-aware records with a source language greater than `0` and no translation parent.

Eric walked through the target-language filtering logic. The patch determines the target page, tries to resolve its site, and stops the copy if no site can be found. The team compared this to the already existing behavior for connected translations and discussed whether missing site information should be logged. They decided not to add additional logging in that case, because the same behavior should remain consistent with the already merged code path.

The team also refined the warning messages for unsupported target languages. André pointed out that messages referring to a "translation of record" were not precise for standalone non-default language records, so Eric changed the wording toward "non-default language record." They also discussed that `pages` records cannot exist in Free Mode. André suggested excluding `pages` early, which Eric accepted because it avoids unnecessary language-awareness checks and reflects that translated pages are always connected to a default-language page.

The team compared the new block with existing copy logic for connected translations and noticed duplicated target page handling. Eric considered whether to reuse an already computed `targetPid` and whether comments in the surrounding code were still useful. André preferred avoiding comments that merely restate the code, and the team kept the focus on functional behavior rather than broader refactoring.

Eric concluded that the patch was functionally plausible, but that the CSV adjustments and Free Mode test relocation still needed work before a new patch set should be uploaded. He planned to finish the test changes, ensure the tests are green, and then share the patch in the channel for further review.
