## Topic 2: Non-Language-Aware `IRRE` Children and Merged Core Patch

Eric returned to the earlier discussion about [patch #88837](https://review.typo3.org/c/Packages/TYPO3.CMS/+/88837), which fixes localization of children for non-language-aware tables. He noted that the patch had been merged on April 11, 2026, after comments from Markus and Lolli. The team recalled the earlier concern about data economy and duplicated records, especially in the context of `IRRE` child records.

Eric summarized the merged behavior as aligned with the initiative's position. When a localized parent record has an `IRRE` field referencing a non-language-aware child table, the child is skipped in the localization branch of the copy process and then handled through synchronization instead. This avoids stealing the original child through remapping and results in separate child records being assigned to the localized parent. Eric interpreted this as the desired outcome: non-language-aware children arrive in target languages as synchronized records belonging to the localized parent.

## Topic 3: TYPO3 v15 Roadmap and Preconditions for Removing Language All

The team discussed the timing after the TYPO3 v14 release and looked ahead to TYPO3 v15 and v16. André emphasized that this is the right time to think about breaking changes before the roadmap and project boundaries become fixed. Eric and André inferred from the established release rhythm that TYPO3 v15 is expected as an October release in 2027, while the relevant development and stabilization windows require decisions much earlier.

Eric framed the central strategic question as whether the initiative can realistically remove `sys_language_uid = -1`. André said that a cleanup of language all needs a roadmap from Lolli or at least alignment on what should be achieved. Eric agreed that the initiative needs full test coverage for all relevant language all use cases before functional changes can be introduced safely.

The team distinguished between deprecating the current mechanism and actually providing an alternative. Eric explained that deprecation only becomes meaningful once TYPO3 has another way to express language all behavior, namely maintaining content once and having it affect all languages without storing `-1` in `sys_language_uid`. The team agreed that the work should continue even if it is not yet clear whether the full replacement can be achieved by the end of the year.

Eric noted that DevDays in August should be used for direct conversations with Core Team members about possible breaking changes. Until then, the initiative should prepare concrete options and a step-by-step plan instead of waiting for others to define the path.

## Topic 4: Free Mode Copy and Move Patches

Eric explained that his newest Free Mode patches are directly related to language handling, but that they are currently not the highest strategic priority compared to removing `sys_language_uid = -1`. The work remains useful because Free Mode behavior is visible to editors and is relevant in a current client project.

The team discussed [patch #93752](https://review.typo3.org/c/Packages/TYPO3.CMS/+/93752), which skips overlays when copying into a Free Mode language. Eric described it as equivalent to earlier copy patches: when content is copied into a non-default language target, all existing translations have to be discarded because a relation from one non-default language to another is not meaningful. Keeping those translations would create orphaned translation records. Eric also mentioned a related `EXT:dbdoctor` cleanup idea based on [`EXT:dbdoctor` PR #171](https://github.com/lolli42/dbdoctor/pull/171), so existing installations can later be checked for legacy orphaned records.

The second patch discussed was [patch #93819](https://review.typo3.org/c/Packages/TYPO3.CMS/+/93819), which guards translated `tt_content` Free Mode moves. Eric noted a remaining UX detail: the existing copy dialog shows the title or type of the affected element, while his move dialog currently only shows "Move record" as the title. Aligning this would require changes in both `paste.ts` and `drag-drop.ts`, because the required title information is already available in one flow but not the other.

Eric was more concerned about the size of the `DataHandler` changes in the move patch. The patch added several hundred lines and removed fewer than one hundred, which makes it hard to review. André warned against blind trust in AI-generated code. Eric planned to let different AI tools review the change against each other first to reduce the patch to the smallest viable `DataHandler` change before asking humans to review it in detail.

Eric explained that the patch introduces many `sys_language_uid = -1` tests as a useful side effect of fixing parent-child language consistency. The concrete bug is that when a parent record's language is changed and new children are added before saving, the new children previously received the old parent language instead of the requested new language.

## Topic 6: Parent-Child Language Rules for `IRRE` and Language All

The team then discussed the conceptual rules for `IRRE` parent-child structures involving `sys_language_uid = -1`. Eric and André agreed that translated child records can make sense below a language all parent: the parent is available in all languages, while the children are rendered only in their own languages. This can model cases where a shared parent record contains language-specific child content.

Eric questioned whether a language all child below a translated parent is meaningful. His reasoning was that a translated parent is only rendered in its own language, so a child that claims to be available in all languages would not actually gain language-spanning output in that context. The team narrowed the discussion to `IRRE` children, because `IRRE` records are considered integral parts of their parent and are deleted with the parent.

Eric derived a possible rule: a connected translated parent should not have its own independent children, because those children should follow the default-language parent. If a translated parent does have its own children, that implies a Free Mode situation. In such a Free Mode translated parent, a language all child appears conceptually unnecessary because the parent itself only exists for one language. By contrast, a default-language parent can reasonably have default-language children or language all children, because the default parent can still be translated and language all children may be intended to appear across those translations.

André agreed that the reasoning sounded plausible, but emphasized that tests are needed to understand what the Core currently allows and which behavior should be preserved or changed. Eric concluded that, for the path toward removing `sys_language_uid = -1`, the focus should stay on test coverage and explicitly documented scenarios. Functional enforcement of parent-child language rules may be wasted effort if the eventual replacement synchronizes language all records into explicit language records anyway.
