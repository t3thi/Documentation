## Topic 1: Cross-Translation Consistency for CType and colPos

At the beginning of the meeting, Eric summarized a chat discussion around Cyrill's question whether `CType` should remain consistent across translations with the `CType` of the default language. The team confirmed that this behavior is intended and recommended keeping it that way.

Eric referred to [patch #85978](https://review.typo3.org/c/Packages/TYPO3.CMS/+/85978), which ensures that both `colPos` and `CType` remain aligned across translations. André emphasized that this is especially important for container elements. If a translated element could switch from a container to a text-media element while also changing `colPos`, the resulting layout and editing behavior would become very hard to reason about. The team noted that projects can still opt out by resetting `l10n_mode=exclude` where they intentionally need divergent `CType` values.

## Topic 2: DataHandler Language All Guard Research and Paste Semantics

The main topic was Eric's research document for a `DataHandler` finding around `sys_language_uid = -1`, following up on the broader [Language All WIP patch #92267](https://review.typo3.org/c/Packages/TYPO3.CMS/+/92267). The relevant guard was located around line 3281 of the current main branch. Eric had used AI-supported analysis and manual review to trace code paths that reach this guard.

The document uses GitHub permalinks based on a concrete commit hash so the referenced source lines remain stable even when the branch changes later. Eric explained that the document was useful not only for the specific `-1` finding, but also for understanding how editorial backend operations travel from UI actions through JavaScript, backend controllers, and finally into the `DataHandler`.

Eric walked through the relevant paste flow. A backend editor first fills the clipboard by copying or cutting a content element. The Layout Module then exposes paste actions through `Paste.ts` and `Page.ts`. The JavaScript code builds a paste request from the target context, while the clipboard still contains the source record information.

The important distinction is that the target context can carry language `0` because it is derived from the closest DOM element, while the source record in the clipboard can still have language `-1`. The `SimpleDataHandlerController` combines the request information with the clipboard contents and builds the `DataHandler` command map. The guard in `process_cmdmap()` then restores language all semantics by setting the target language back to `-1` when the source record is a language all record.

The team clarified that the Layout Module paste flow is a highly relevant browser-originated producer of this runtime form, but it is not necessarily the only possible entry point. Eric's analysis indicated that the guard needs a language-aware source row from the database, an extended paste command with an updated language field, and a requested downgrade from source language all to target language `0`.

Eric explained that direct live copy scenarios are already covered by existing regression tests. The missing coverage is mainly in Workspace scenarios that execute the same extended paste path. Move flows appear to be less critical because moving a record tends to preserve the record's own fields directly, while copy flows have to re-materialize the target command data.

## Topic 3: Workspace Copy Tests for Language All Content

Eric showed a prepared local patch that had not yet been uploaded and adds four missing Workspace tests for copying language all content. The affected functional data-scenario areas are Workspace Discard, Workspace Modify, Workspace Publish, and Workspace Publish All. In each case, a new `copyLanguageAllContent` scenario is added to existing `ActionTest` classes.

The team inspected how existing helper methods build record data for copying a language all content element. Eric noted that the target language for a `-1` element can be `0` or `-1` depending on the paste position, for example whether the element is pasted near another language all element. This confirmed that the existing helper comments were meaningful.

The team looked more closely at the Workspace Discard case. The test copies a language all content element inside a workspace and then clears or discards the workspace record. André explained that this represents reverting the workspace change. Eric compared the expected CSV result with the import dataset and reasoned that a discarded workspace record should disappear from the database rather than remain as a deleted placeholder.

Eric found the discard scenario difficult as an entry point for understanding Workspace behavior and suggested that Workspace Modify may be easier to inspect next. He decided to hold back the patch for the moment, share the research document in the channel, and continue reviewing the Workspace cases in the next meeting.
