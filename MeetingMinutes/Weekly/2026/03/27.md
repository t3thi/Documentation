## Topic 2: rel Attribute Patches and RTE Link Security (Off-Topic)

André brought up Albrecht Köhnlein's [patch #91146](https://review.typo3.org/c/Packages/TYPO3.CMS/+/91146), which relates to Forge [issue #107771](https://forge.typo3.org/issues/107771) and adds TypoScript configuration for default `rel` attribute values on external links. He had tested the patch and found that changing the configured security value did not affect `RTE` links as expected.

Eric explained that his own recently merged [patch #93049](https://review.typo3.org/c/Packages/TYPO3.CMS/+/93049) focused on non-RTE links, because `RTE` links already have their own configuration path. André showed that the tested patch adds a `config` option for link security values, but that existing `ATagParams` data can already contain `rel="noreferrer"`, so the new configuration may not override the value in the tested scenario. Eric suspected that the patch might be applied too late in the rendering pipeline for the behavior André expected.

The meeting ended with a live investigation of André's `rel` attribute finding. Eric attempted an AI-assisted review, but the first analysis misunderstood part of the issue. André and Eric then tested manually and distinguished between non-RTE links, menu links, call-to-action links, and `RTE` links.

They found that the patch can work for a non-RTE call-to-action link, where the link result receives `rel="noopener"`. For `RTE` content, however, the database already contained `rel="noreferrer noopener"` after the `CKEditor` processing path. Eric connected this to CKEditor's `addTargetToExternalLinks` behavior, which is configured as `false` in TYPO3 and can cause the editor to write security-related `rel` attributes itself. The team concluded that this explains why the global TypoScript setting may appear ineffective for `RTE` links and that this caveat should be mentioned in the Gerrit discussion.

## Topic 3: Non-Language-Aware IRRE Children and Synchronization Strategy

Eric then returned to a T3THI-relevant Core discussion about [patch #88837](https://review.typo3.org/c/Packages/TYPO3.CMS/+/88837), which fixes localization of children for non-language-aware tables. The patch concerns language-aware parent records with `IRRE` children that are themselves not language-aware. The key question is whether such children should be copied or synchronized per translated parent even though the child table has no own language concept.

Eric summarized the debate with Torben and Markus. Torben had raised concerns about database economy and unnecessary duplicate records. Eric and André argued that deliberate denormalization can simplify output logic and avoid complex lookups, especially for `IRRE` structures where child records belong to their parent. André stated that if such child records are not meant to be duplicated with their parent, the relation model should be changed to a normal relation instead of `IRRE`.

The team concluded that the specific patch may not be the best place to settle the broader concept. Eric said the initiative needs a reliable discussion with Core maintainers such as Torben, Markus, Lolli, and Benni about whether database-level synchronization during localization is acceptable. This decision affects much of the initiative's thinking about translation handling.

André expected that the patch may remain unresolved until such a discussion happens in person. Eric noted that the upcoming DevDays could become important for aligning with Core maintainers, especially because the initiative has already produced several concrete patches and more test coverage since the beginning of the year.

## Topic 5: Recent Core Features, Site Set Boundaries, and Form Storage (Off-Topic)

Eric summarized several recently merged or discussed Core features. His merged work included [localized enum labels in settings definitions](https://review.typo3.org/c/Packages/TYPO3.CMS/+/92902), [extraction of remaining Core enum labels to XLIFF files](https://review.typo3.org/c/Packages/TYPO3.CMS/+/93041), [new events for modifying Indexed Search result sets](https://review.typo3.org/c/Packages/TYPO3.CMS/+/92578), and [a setting to switch Indexed Search to sliding-window pagination](https://review.typo3.org/c/Packages/TYPO3.CMS/+/92896). The team briefly discussed whether a future "Load More" pagination would need AJAX to be meaningful.

The team also touched on rejected or constrained Site Set ideas. Eric mentioned that [YAML imports for settings definitions](https://review.typo3.org/c/Packages/TYPO3.CMS/+/92516) were rejected because Site Sets are intended to remain self-contained. André added that environment placeholders are also intentionally disabled in settings definitions, which can be inconvenient but follows the same containment idea.

André asked about the direction of TYPO3 form storage after seeing database-backed form storage discussed. Eric distinguished between file-based forms stored in `fileadmin`, which do not belong there conceptually, and forms shipped by an extension or site package, which should remain possible. They agreed that both file-based presets and database-backed editor-managed forms have valid use cases depending on whether a form is stable site-package configuration or changing editorial content.

Eric also noted that the Camino [language navigation patch #92909](https://review.typo3.org/c/Packages/TYPO3.CMS/+/92909) had been merged as a task rather than as a feature. Since `theme-camino` itself is still being built up, this classification was considered acceptable.
