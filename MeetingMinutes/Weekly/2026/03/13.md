## Topic 1: Language Switching in Product Data

After a brief personal exchange, André described a recent project case where German and English had to be swapped across a product extension. The team discussed that it was safer to swap the translated text fields in several tables instead of exchanging `sys_language_uid` values. André explained that keeping the UIDs stable avoided breaking references, image relations, and other table links.

Eric noted that this kind of practical case illustrates why better future tooling around language handling would be valuable. André added that AI support helped with the repetitive table work, although he still reviewed the individual steps carefully, especially for the first table.

## Topic 2: Orphaned Translation Warnings and Test Coverage

Eric reviewed the current state of [patch #93063: Show warnings on orphaned translations](https://review.typo3.org/c/Packages/TYPO3.CMS/+/93063), which relates to Forge [issue #108358](https://forge.typo3.org/issues/108358). The patch adds specific warnings in the Layout Module for translated content elements whose `l10n_parent` points to a missing record, a non-default language record, or a record in language all. These invalid parent references are treated as connected translations when determining the language mode, and they also trigger the generic stale translation flash message.

The team confirmed that mixed mode must still only be used when standalone and connected translations coexist. Invalid translation parents alone should not trigger mixed mode. Eric showed that the patch had been updated according to André's earlier review comments and that the wording now refers to the Layout Module rather than the Page Module.

Eric walked through the updated functional tests. The patch now verifies the warning labels, checks that the generic stale translation flash message appears for invalid translation parents, and ensures that the message is only emitted once when invalid translation parents and mixed mode appear together.

André found the tests plausible after the updated condition was split into two logical blocks: one for determining mixed mode and one for triggering the stale translation message. Eric noted that Oliver had already verified the tests, which gave additional confidence, even though Oliver may not have reviewed all code details.

## Topic 3: Typolink rel Attribute Handling (Off-Topic)

Eric briefly asked for feedback on [patch #93049](https://review.typo3.org/c/Packages/TYPO3.CMS/+/93049), a separate Core patch that adds support for a sixth Typolink parameter to store the `rel` attribute outside the `RTE` context. He explained that Core already assumed `rel` support in several places, but non-RTE link fields previously had no consistent input and rendering path for it.

André was cautious about the additional Typolink parameter because the positional Typolink format is already complex. The team compared this approach with alternatives such as reusing an existing parameter for arbitrary attributes, but Eric noted that changing the meaning of the second parameter would be much more likely to break existing parsers and extensions.

## Topic 4: Language All Findings, Test Coverage, and DataHandler Paths

The team returned to the [WIP patch #92267](https://review.typo3.org/c/Packages/TYPO3.CMS/+/92267) that had marked uses of `sys_language_uid = -1` throughout TYPO3 Core. Eric summarized Benni's earlier suggestion to centralize the special value in a class constant. André said that if a constant is introduced for `-1`, a corresponding constant for `0` should be considered as well, because many checks distinguish language all from the default language.

Eric and André agreed that constants would make direct occurrences easier to find with an IDE, but they would not fully solve the underlying semantic problem. Some code uses implicit checks such as greater-than comparisons, and some call sites really need higher-level methods such as checking whether a value means language all or default-language scope. The team therefore distinguished between replacing literal values as an incremental step and later introducing semantic abstractions.

André emphasized that Lolli should be involved before the team decides how `-1` should ultimately be removed. Eric recalled Lolli's earlier position that test coverage must come before functional changes. The team therefore agreed that the current findings should primarily be used to identify missing tests.

Rather than preparing one large patch that covers all findings, André recommended smaller, mergeable patches grouped by use case or extension. Eric agreed that the first concrete area should be `DataHandler`-related behavior, because the initiative already has experience writing functional CSV-based tests for `DataHandler` scenarios.

André suggested a deterministic way to determine which existing tests already hit a relevant code path: insert a deliberate failure at the `DataHandler` location under investigation and run the test suite. If existing tests fail, the team can identify which scenarios already cover the path. If all tests remain green, the missing coverage is clear.

Eric liked the approach and proposed applying it iteratively. The first step would put the failure directly inside the relevant `-1` branch. Later steps could move the failure outward into surrounding code to discover adjacent flows that are still relevant to language all handling. This would reduce unnecessary AI analysis and provide concrete test data for subsequent review.
