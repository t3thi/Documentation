## Topic 1: TYPO3 v15 Roadmap and Product Strategy Contacts

Eric opened with the current state of the [TYPO3 CMS development roadmap](https://typo3.com/typo3-cms/development-roadmap/roadmap). The available roadmap was still focused on TYPO3 v14, while TYPO3 v15 was not yet represented. He described the TYPO3 Product Strategy Group and the board responsibility for product strategy as the likely entry point for bringing translation handling into the next roadmap.

The team discussed possible contacts. Eric identified Martin Helmich as the likely contact for product strategy operations, while Benni was seen as relevant for technical feasibility, Core Team perspective, and mergeability. André noted that Benni's role appeared to be translating strategic wishes into technical terms rather than driving those wishes as the primary decision maker. Eric concluded that the strategic roadmap is not decided by a single person, but through the board and the product responsibility within the TYPO3 Association structure.

Eric suggested approaching Martin with the initiative's goal, which had originally been discussed for TYPO3 v14 and had now shifted toward TYPO3 v15. André proposed phrasing the goal as further development of multilingual support. The team preferred this broader framing over a narrow promise, because it leaves room for several related improvements while still describing the product area clearly.

## Topic 2: Roadmap Framing for Language All Removal

The team examined how the removal of `sys_language_uid = -1` can be explained from an editor's point of view. André stated that simply removing language all does not create an immediate editor-facing benefit and could even look like a feature being taken away. Eric agreed that removal is only viable together with a migration path and an alternative way to maintain content once and distribute it across languages.

Eric described the previously discussed alternative as a boolean flag on the default-language level that marks a record for synchronization into all languages. A later step could turn this into a multi-select for explicit target languages, which would create a more visible product feature: editors could decide into which languages a record should be synchronized instead of being limited to all-or-nothing language all behavior.

Eric framed `sys_language_uid = -1` removal as foundational work whose value becomes easier to explain through later capabilities, including more explicit synchronization targets and a possible long-term move toward `BCP 47` based language identification. The team noted that this alone is still a weak roadmap story compared with changes that editors can experience directly.

## Topic 3: `CType` and `colPos` Lock in Connected Mode

André related the current discussion to multilingual projects that had intentionally used connected translations with different content-element types. He noted that moving such cases into Free Mode can cost useful connected behavior. The concrete Core reference was [patch #85978](https://review.typo3.org/c/Packages/TYPO3.CMS/+/85978), which resolved [Forge ticket #60357](https://forge.typo3.org/issues/60357) and disabled custom `colPos` and `CType` values for connected translations.

Eric and André referred to the current Slack discussion triggered by Mathias Brodala's question why `CType`, and not only `colPos`, had been locked. Mathias described a project use case in which the default language contains teaser elements linking to pages that intentionally have no translations, while the translated records used different content elements with textual fallback content and links back to the default-language page. The team treated this as an example of the broader tension between project-specific Connected Mode workarounds and the initiative's model, where changing the nature of a content element is structural divergence that fits better with Free Mode.

The team also noted the related documentation patch [#93853](https://review.typo3.org/c/Packages/TYPO3.CMS/+/93853), which resolves [Forge ticket #109718](https://forge.typo3.org/issues/109718) and adds documentation for the `CType` and `colPos` lock on connected translations. The discussion reinforced that the upgrade wizard and the default behavior may be technically consistent, but can still create substantial project work when installations had intentionally used different translated `CType` values.

## Topic 4: Hidden Structural Layer for Flexible Translations

Eric connected the need for more flexible translated content to the older shadow-record idea. He shared the 2022 discussion thread ["Rethinking translation handling based on a session of T3CMD 2022"](https://talk.typo3.org/t/rethinking-translation-handling-based-on-a-session-of-t3cmd-2022/5515) and recalled that the initiative originally started from a similar concept: the default language could become a hidden structural layer rather than a visible editorial language. Existing installations already sometimes use the default language as a non-output structure layer, but editors still see a visible column that is not actually useful for content work.

André sketched a variant that would keep the historical technical model with `sys_language_uid = 0` as a structural layer, while every real editorial language, including the current default output language, receives its own positive language identifier. When an editor adds content in a translation, the structural layer would receive the necessary structural record in the background, so Connected Mode can be preserved while the actual language content remains flexible.

Eric argued that the concept should stay close to the current storage model. He rejected reversing the relation so that the structural record would store a list of target languages, because that would introduce a more complex one-to-many relation or relation table at a central point. He preferred building on existing synchronization concepts such as `l10n_mode=exclude`, `allowLanguageSynchronization`, and a possible future `enforceLanguageSynchronization`.

### Topic 4.1: Page Tree and Page Module UX for Missing Translations

The team explored how a hidden structural default layer would appear in the backend. Eric and André agreed that the structural layer would mainly contain nodes, not translated editorial content. The Page Tree would therefore need to show pages in the currently selected editing language and also represent pages that exist structurally but are not yet translated into that language.

André suggested that missing pages could be shown in a subdued visual style. If the current language is English and a page only exists in Spanish or German, the Page Tree could still show a placeholder at the correct structural position, but mark it clearly as not available in the current language. Eric noted that TYPO3 currently uses configured language icons, often flags, as visual identifiers. The team considered reusing the existing identifier for now while treating the broader flag discussion as a separate topic.

The team discussed whether the Page Tree needs to show all languages in which a page already exists. André argued that the Page Tree mainly needs to communicate structure and the fact that a page exists somewhere, while detailed language availability belongs in the Page Module, List Module, or language comparison view. Eric agreed that showing many language icons in the tree would quickly become crowded.

For the Page Module, André proposed avoiding a confusing situation where clicking a missing translation would show content from another language as if it belonged to the currently selected language. Instead, the Page Module should show a clear empty-state message that the page is available in other languages and offer a translation action. Eric compared this to TYPO3's existing page translation dialog, where an editor selects the source language and then chooses which content elements should be translated.

The team concluded that a generic message with a `Translate Page` or `Translate Now` button would be sufficient. The editor already knows the target language from the selected editing context. After clicking the button, the existing translation dialog can ask for the source language and content selection.

### Topic 4.2: Editing Language Selector and Prototype Planning

Eric and André refined the terminology for the language context. They rejected "source language" because that sounds like the language from which a translation is copied. They settled on "editing language" as the more accurate term, because it describes the language currently used as the basis of the editing interface. André compared this to a high-level context switch similar to Workspaces, while Eric emphasized that this must not change the backend UI language itself. A Spanish editor should still be able to use Spanish backend labels while editing American English content.

In the language comparison view, the selected editing language could become the primary column. If a page does not yet exist in that language, the column would show the translation prompt instead of editable content. Other languages could still be shown as comparison or source columns, depending on the selected workflow.

The team then discussed how to communicate the concept to the Product Strategy Group and the Core Team. André suggested preparing sketches or a clickable prototype that reflects the TYPO3 v14 Layout Module, the List Module, and the Page Tree closely enough for others to understand the intended direction. Eric suggested using AI-assisted prototyping or manual browser and screenshot editing rather than preparing a real Core patch at this stage.

By the end of the meeting, the team had shifted the near-term focus from patching to strategic visualization. The proposal should present editing-language selection, more flexible translated content in Connected Mode, and the hidden structural layer as the visible product story. Technical implementation details can then be discussed with Benni and other Core contributors once the intended editor experience is easier to understand.
